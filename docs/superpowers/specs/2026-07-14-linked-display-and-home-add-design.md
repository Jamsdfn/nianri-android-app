# App 与小部件展示历法联动设计

## 目标

将新历/农历的“展示选择”从 App 与各小部件独立保存，改为按重要日子统一联动；同时移除首页右下角重复且遮挡内容的悬浮新增按钮。

此变化只统一展示状态。日子创建时选择的新历或农历仍是永久倒计时基准，切换展示不会改变倒计时日期计算方式。

## 展示状态模型

每个重要日子的 `appDisplay` 成为唯一有效展示状态：

- App 首页切换某个日子时，更新该日子的 `appDisplay`，并刷新所有引用该日子的小部件。
- 任一小部件切换时，更新它所引用日子的 `appDisplay`，并刷新 App 数据流以及所有引用同一日子的小部件。
- 引用其他日子的小部件不受影响。
- 编辑页保存已有日子的展示选择后，既有的全量小部件刷新流程会读取新的统一状态。

小部件偏好表中的 `display` 字段暂时保留，以避免数据库迁移和升级风险。写入统一状态时同步更新该字段，但小部件渲染必须以重要日子的 `appDisplay` 为准，因此升级前已经不一致的旧记录也会立即恢复一致。

## 数据与刷新流程

在一个 Room 事务内完成以下操作：

1. 找到目标日子或目标小部件引用的日子。
2. 计算或接收新的展示历法。
3. 更新重要日子的 `appDisplay`。
4. 更新所有引用该日子的小部件偏好记录中的兼容字段。
5. 返回所有受影响的 `appWidgetId`。

事务提交后逐个刷新受影响的小部件。单个刷新失败时沿用现有聚合错误策略；数据库中的统一展示状态不会回退为两份互相冲突的值。App 首页保留当前乐观切换与失败提示行为。

## 首页新增入口

- 删除首页右下角 `home-fab` 按钮。
- 保留标题栏右上角 `home-add` 按钮。
- 保留空状态里的“新建日子”入口，避免空列表时只有较远的标题栏入口。
- 首页列表底部留白从为悬浮按钮预留的 104dp 调整为普通安全留白；Snackbar 也不再为悬浮按钮额外上移。

## 测试与验证

- 仓库测试：从任一小部件切换后，日子、所有同日小部件兼容字段和返回的受影响 ID 一致更新。
- 仓库测试：从 App 设置展示后，仅同日小部件同步，其他日子不变。
- 展示测试：小部件解析忽略升级前不一致的偏好字段，以日子的 `appDisplay` 为准。
- 控制器测试：小部件切换刷新所有同日实例，不刷新其他实例。
- HomeViewModel 测试：App 切换调用联动更新入口并保持现有失败回滚。
- HomeScreen 测试：`home-add` 存在且可用，`home-fab` 不存在，空状态入口保留。
- 在 API 26、31、36、37.1 模拟器运行完整测试，并覆盖安装到小米 15 Pro 真机验证双向联动。
