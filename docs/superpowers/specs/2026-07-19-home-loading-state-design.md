# 首页本地数据加载态设计

## 背景

应用启动时，`HomeViewModel` 当前以空列表作为日子数据流的初始值。Room 返回第一次真实查询结果前，首页无法区分“数据尚未加载”和“加载完成但没有日子”，因此会短暂显示新建引导，让已有数据的用户误以为数据丢失。

## 目标

- 启动读取本地日子期间显示明确、简洁的加载反馈。
- Room 首次返回前不显示新建引导、日子列表或历法说明。
- Room 首次返回后，根据真实数据展示已有日子列表或空状态。
- 不增加固定等待时间，不拖慢实际启动速度。

## 状态设计

`HomeUiState` 增加 `isLoading: Boolean`，初始值为 `true`。

`HomeViewModel` 将日子数据流包装成具有“等待首次结果”和“已加载”语义的内部状态。第一次收到 Room 数据后，`isLoading` 变为 `false`；此后继续沿用现有列表投影、置顶、日期展示切换和错误处理逻辑。

加载完成后的空列表是有效结果，必须显示现有的新建引导。加载中的空列表只作为内部占位，不得触发空状态。

## 界面设计

加载期间保留首页深色渐变背景和“念日”标题，以维持稳定的应用识别：

- 隐藏右上角“＋”新建按钮。
- 隐藏历法说明、日子卡片、分组标题和新建引导。
- 在内容区域中央显示紫色圆形进度指示器。
- 在指示器下方显示“正在加载日子…”。

首次数据到达后，加载内容直接切换为现有首页内容，不额外设置最短展示时长或过渡延迟。

## 数据流

1. `HomeViewModel` 创建，首页状态为加载中。
2. `HomeScreen` 只显示标题和加载反馈。
3. Room 发出第一次日子列表。
4. `HomeViewModel` 将状态标记为加载完成并投影列表。
5. 非空列表显示日子卡片；空列表显示新建引导。

## 错误处理

本次改动不扩展仓库错误策略。现有 Room 本地流正常返回时进入加载完成状态；日期投影失败仍按现有逻辑显示“日期暂不可用”。未来如需处理数据库流异常，应另行设计可重试的首页错误态。

## 测试

- ViewModel 初始状态必须为加载中，且第一次数据发出后变为加载完成。
- Compose 加载态必须显示进度反馈，不显示新建引导与新建按钮。
- 加载完成且列表为空时，必须显示现有新建引导。
- 现有非空首页、日期展示切换和错误提示测试继续通过。

## 非目标

- 不增加独立启动页或固定加载时长。
- 不实现骨架屏、闪烁动画或跨页面加载组件。
- 不修改 Room、数据库结构或本地数据格式。
