# 每个日子自定义提醒时间设计

## 目标

每个重要日子可以单独设置提醒时间。新建日子的默认时间为本地 09:00；现有日子升级后也保持 09:00。该时间统一用于当天固定提醒以及用户开启的提前 14、7、3 天提醒。

## 数据模型与迁移

- 在 `ImportantDay` 和 `ImportantDayEntity` 中增加 `reminderTimeMinutes`，表示从当天 00:00 起经过的分钟数。
- 合法范围为 `0..1439`；09:00 对应 `540`。
- Room 数据库从版本 1 升级到版本 2，通过迁移为 `important_days` 增加非空整数字段，默认值为 `540`。
- `AppContainer` 显式注册 1→2 迁移，不使用破坏性迁移，因此已有日子、提醒选项及小部件关联不会丢失。
- Repository 在保存时校验分钟值，并负责实体与领域模型之间的映射。

选择单个分钟字段而不是分别保存小时和分钟，可以原子地表达一个有效时间，减少字段不同步的可能。提醒时间不放入 SharedPreferences，避免与日子的删除、备份和恢复生命周期分离。

## 编辑与展示交互

- `EditDayUiState` 保存 `reminderTimeMinutes`，新建状态默认 `540`，编辑已有日子时从数据库加载。
- 编辑页提醒区域提供可点击的时间项，文案展示为“提醒时间 HH:mm · 固定开启”。
- 点击时间项打开 Android 系统时间选择器，支持小时和分钟，并跟随手机的 12/24 小时制。
- ViewModel 接收选择结果，将小时和分钟转换为分钟值；非法输入直接拒绝，不改变当前状态。
- 提前 14、7、3 天的开关保持现有行为。自定义时间统一作用于当天和所有已开启的提前提醒。
- 详情页提醒摘要始终包含当天提醒及时间。例如没有提前提醒时显示“当天 08:35”，开启 14、7 天时显示“当天 08:35；提前 14、7 天”。

## 调度与补发

- `AndroidReminderScheduler` 不再固定调用 `atTime(9, 0)`，而是把 `reminderTimeMinutes` 转换为对应的本地小时和分钟。
- 当天提醒和提前提醒使用同一个本地时间，并继续按当前系统时区转换为精确闹钟。
- 保存或重建提醒时，若本次发生日就是今天且已达到该日子的自定义时间，则沿用现有即时派发路径补发当天提醒。
- 若尚未达到自定义时间，则安排未来闹钟，不提前补发。
- 时区、系统日期或时间变化后的重建逻辑保持不变，但会使用每个日子各自保存的时间重新计算。
- 当天提醒去重账本保持不变，同一日子的同一次发生日期最多成功发送一次。

## 错误处理与兼容性

- UI 通过系统时间选择器产生合法值，ViewModel 和 Repository 仍分别守住 `0..1439` 的边界。
- 迁移使用数据库默认值 `540`，即使用户升级后尚未编辑任何日子，也能继续在 09:00 收到提醒。
- 精确闹钟权限不足、通知权限不足及通知发送失败继续沿用现有提示、审计和重试行为。
- 此变更不修改 `reminderMask` 的含义，也不改变当天提醒固定开启的规则。

## 测试范围

- 领域与 Repository 测试：新建默认 09:00、自定义值保存读取、非法分钟拒绝。
- 数据库迁移测试：v1 数据升级到 v2 后各字段保留，提醒时间为 `540`。
- ViewModel 测试：加载已有时间、修改小时分钟、保存自定义值、拒绝非法输入。
- UI 测试：时间项展示、点击打开选择器、选择后更新文案，并保持提前提醒开关可独立操作。
- 调度测试：当天及提前提醒都在自定义时间触发；09:00 默认行为不回退；自定义时间前不补发、达到或超过时间后当天补发。
- 详情测试：摘要始终显示当天提醒时间，并正确拼接可选提前提醒。

## 不在本次范围内

- 全局统一提醒时间设置。
- 为当天提醒和不同提前提醒分别设置多个时间。
- 关闭当天固定提醒。
- 秒级精度、重复响铃或稍后提醒。
