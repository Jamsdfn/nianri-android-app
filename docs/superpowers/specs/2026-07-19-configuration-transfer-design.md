# 念日配置迁移设计

## 背景与目标

念日目前把纪念日数据保存在本机。用户更换设备时，需要在 A 设备导出全部纪念日配置，并在 B 设备直接导入后继续使用。

本功能提供单文件导出与合并导入。它迁移纪念日的业务配置，同时保留 B 设备已有数据，并明确处理重名和置顶冲突。

## 范围

导出以下纪念日字段：

- 名称
- 倒计时历法
- 月、日
- 应用内日期展示历法
- 提前提醒选项
- 提醒时间
- 置顶状态

以下设备相关数据不导出：

- 数据库 ID、创建时间和更新时间
- 桌面小部件实例及其绑定
- 系统通知权限、精确闹钟权限
- 设备或应用进程状态

本期不提供云同步、二维码迁移、选择部分纪念日导出或覆盖式恢复。

## 用户界面

首页标题栏右侧在现有 `+` 左侧增加“迁移”文字按钮。点击后打开底部迁移面板。

迁移面板顶部包含“导出”和“导入”两个 Tab，默认选择“导出”。

### 导出 Tab

- 显示将要导出的纪念日数量。
- 主按钮为“导出全部配置”。
- 没有纪念日时按钮禁用，并显示“暂无可导出的纪念日”。
- 点击后使用 Android Storage Access Framework 的创建文档流程，让用户选择保存位置。
- 默认文件名为 `念日配置-yyyyMMdd.nianri.json`，日期使用设备当前本地日期。
- 保存成功或失败后在迁移面板中显示明确结果。

### 导入 Tab

- 说明导入会与本机数据合并，不会删除已有纪念日。
- 主按钮为“选择配置并导入”。
- 点击后使用 Android Storage Access Framework 的打开文档流程选择配置文件。
- 选中文件后立即读取、完整校验并执行导入，不再增加第二次确认。
- 处理期间禁用重复操作。
- 成功后显示导入总数和因重名而改名的数量。
- 失败后显示可行动的错误原因，且不留下部分导入数据。

Storage Access Framework 不要求应用新增广泛存储权限。

## 文件格式

导出文件使用 UTF-8 JSON，建议 MIME 类型为 `application/json`。首版结构如下：

```json
{
  "format": "nianri-configuration",
  "version": 1,
  "exportedAt": "2026-07-19T09:30:00Z",
  "days": [
    {
      "name": "结婚纪念日",
      "basis": "SOLAR",
      "month": 8,
      "day": 6,
      "appDisplay": "SOLAR",
      "reminders": [14, 7, 3],
      "reminderTimeMinutes": 540,
      "isPinned": true
    }
  ]
}
```

`format` 固定为 `nianri-configuration`，`version` 首版为 `1`，`exportedAt` 使用 UTC ISO 8601 时间。`days` 可以为空。每个纪念日对象只包含“范围”一节列出的业务字段，不包含本机数据库标识。

编码器输出稳定、明确的字段名称。解码器必须在写入数据库前验证：

- 格式标识正确
- 版本受当前应用支持
- 所有必填字段存在且类型正确
- 名称去除首尾空白后非空；导入规划和最终保存均使用去除首尾空白后的名称
- 历法枚举受支持
- 月、日与所选历法匹配
- 提醒偏移量均受支持
- 提醒时间位于 `00:00` 至 `23:59`
- 文件中至多有一个置顶项

首版仅接受版本 `1`。未知的新版本返回“配置版本暂不支持”，避免按旧规则误读。

## 导入规则

导入采用合并模式，不删除或修改 B 设备已有纪念日。

### 名称冲突

名称冲突按去除首尾空白后的字符串完全相等判断，不做大小写或 Unicode 近似匹配。导入过程中维护一组已占用名称，初始包含 B 设备的全部现有名称；每成功规划一个导入项，立即把其最终名称加入该集合。

如果导入名称未被占用，则保留 A 设备中的原名称。

如果名称已被占用，则改为：

```text
原名称-yyyyMMdd导入
```

其中日期为 B 设备开始执行本次导入时捕获一次的本地日期，同一批数据始终使用该日期。例如 `结婚纪念日-20260719导入`。

若该名称仍被占用，则依次尝试：

```text
原名称-yyyyMMdd导入-2
原名称-yyyyMMdd导入-3
```

直到获得未占用名称。

### 置顶冲突

- 如果 B 设备导入前已有置顶纪念日，B 的置顶保持不变，所有导入项均以非置顶状态写入。
- 如果 B 设备导入前没有置顶纪念日，则保留 A 配置中的置顶状态。
- 文件校验保证 A 配置中至多有一个置顶项，因此导入后仍满足单置顶约束。

### 标识与设备状态

所有导入项由 B 设备生成新的数据库 ID。导入不尝试恢复小部件绑定或权限状态。

## 原子性与副作用

文件先完整读取、解析、校验并生成导入计划，之后才开始写数据库。

全部纪念日通过单次 Room 事务写入。任意一条写入失败时回滚整次数据库导入，不留下半套数据。

事务提交后，应用统一重建全部纪念日提醒并刷新小部件/首页数据。小部件不会自动绑定到新导入的纪念日，但已有小部件会按其原绑定状态刷新。

文件解析或校验失败时不触发数据库写入、提醒重建或小部件刷新。

## 组件边界

### `TransferCodec`

负责业务模型与版本化 JSON 之间的编码、解码和格式级校验。它不访问 Android 文件 API 或数据库。

### `ConfigurationTransferService`

负责读取当前纪念日生成导出快照；在导入时执行名称规划、置顶规划和事务写入；提交后协调提醒重建与界面/小部件刷新。

### `TransferViewModel`

负责迁移面板状态，包括当前 Tab、导出数量、处理中状态、成功统计和错误文案。文件内容通过清晰的输入/输出边界交给服务，不把 Android `Uri` 传入领域层。

### `TransferSheet`

负责 Compose 底部面板和 Tab UI。导航层持有系统创建/打开文档 launcher，并把读写结果交给 ViewModel。

## 错误处理

用户可见错误至少区分：

- 选择的文件不是念日配置
- 配置文件已损坏或结构不完整
- 配置版本暂不支持
- 纪念日字段非法
- 文件无法读取或写入
- 数据库导入失败
- 导入成功但提醒刷新失败

用户取消系统文件选择不视为错误，不显示失败提示。

数据库提交后的提醒刷新属于独立副作用：如果提醒刷新失败，已成功导入的数据仍保留，并提示用户重新打开应用或检查提醒权限。此情况不能回滚已经提交的数据库事务。

## 测试策略

### 单元测试

- 版本 `1` JSON 编码/解码往返保留全部业务字段。
- 拒绝错误格式标识、未知版本、缺失字段、非法枚举、非法日期、非法提醒和多置顶文件。
- 无重名时保留原名称。
- 与 B 设备现有名称重名时追加导入日期。
- 日期后缀仍重名时追加递增序号。
- 同一导入文件中的名称冲突也使用相同规划规则。
- B 已有置顶时不覆盖；B 无置顶时继承 A 的置顶。
- 任意一条写入失败时事务回滚。

### 集成测试

- 导入后所有数据获得新 ID，且业务字段正确。
- 事务提交后重建提醒并刷新界面/小部件。
- 解析或校验失败时没有数据库或副作用调用。
- 导出文件能够被新安装的应用实例成功导入。

### UI 测试

- 首页“迁移”按钮位于 `+` 左侧并可打开底部面板。
- “导出 / 导入”Tab 可切换，默认选中“导出”。
- 处理中按钮禁用，完成后显示成功统计或明确错误。
- 取消文件选择时界面保持可操作且不显示错误。

## 验收标准

- A 设备可把全部纪念日业务配置导出为一个 `.nianri.json` 文件。
- B 设备可选择该文件并合并导入，不需要存储权限。
- B 设备已有纪念日不被删除或修改。
- 重名、置顶和序号兜底规则与本文一致。
- 非法或不兼容文件不会造成部分写入。
- 导入后提醒按 B 设备当前权限重新建立，首页立即显示导入结果。
